## TL;DR

Ships `sdlc-workflows` — a plugin that lets a Claude Code user dispatch delegated SDLC tasks (design → implement → review, sequential or parallel, with cycles) to isolated Docker containers orchestrated by Archon, and monitor them from the same terminal. 108 commits, ~38k LoC added, 276 unit tests + the integration smoke suite pass (see matrix below). Closes EPIC #96.

## Summary

- **`sdlc-workflows` plugin** with 7 skills (`workflows-setup`, `deploy-team`, `author-workflow`, `workflows-run`, `workflows-status`, `manage-teams`, `teams-status`) and one specialist agent.
- **Three-tier Docker image model** (`sdlc-worker:base` → `dev-team` / `review-team` / `full`) with per-node agent enforcement via team manifests.
- **Archon orchestration** for sequential and parallel workflows via a bash-node preprocessor (an `image:` field in YAML is rewritten to a `bash: docker run …` node at dispatch time).
- **`loop.stages:` multi-stage cycle primitive** — declarative designer → dev → review cycles with termination-signal detection.
- **Three-tier credential resolver** (macOS Keychain → named volume → config file), with EXIT/INT/TERM cleanup traps.
- **Security hardening** — `--cap-drop ALL`, `--no-new-privileges`, UID 1001, memory/CPU caps, HEALTHCHECK, signal trap inside the container, `shlex.quote` on every user-string, widened rsync exclude list so a prompt-injected worker cannot read `.env`, `.ssh/`, `.aws/`, `.docker/`, etc.
- **Status observability** — `workflows-status` hits REST on `archon serve` if up, falls back to reading `~/.archon/archon.db` directly (works on a fresh machine with no server running). `--run-id` accepts any unique prefix, matching the 8-char form `--recent` prints.
- **Archon-native monitoring surfaced + proven** — workflows-run, workflows-setup, workflows-status, and the quickstart now name the four existing Archon observation paths: live per-node stderr (`[name] Started/Completed`), the **Archon web UI at `archon serve` / http://localhost:3090** (bundled, downloads on first run), CLI snapshot (`archon workflow status`, `archon isolation list`), and `docker logs -f` for per-container detail. `archon workflow run --resume` is now called out for failure recovery. **Each surface is asserted end-to-end:** 9 new SQLite-fixture pytest cases exercise `workflows-status` against a temp Archon-schema DB (runs everywhere, no CLI needed); `run-e2e.sh` gains 4 CLI-level monitoring checks when `archon` is on PATH (workflows-status finds the run, archon workflow status / isolation list succeed) plus 3 opt-in live checks under `--live-monitoring` (archon serve answers REST within 30s, `/api/workflows/runs` contains the real run id, `/` returns HTML proving the web UI is actually served).
- **Full documentation suite** — `CLAUDE-CONTEXT-workflows.md`, quickstart, troubleshooting, `author-workflow` skill (interactive YAML generator).

## Honest "what we got wrong" — pivots during the EPIC

1. **Archon ContainerProvider patch → bash-node preprocessing (Phase B).** The original plan was to land a `ContainerProvider` upstream in Archon (coleam00/Archon#1197). Once we discovered Archon has no per-node executor dispatch, we retired the patch (commit c7383a1) and moved isolation out of Archon entirely. The preprocessor rewrites every `image:` node into `bash: docker run …` so Archon sees a standard node type and we keep control of container lifecycle.
2. **`--read-only` root filesystem dropped (Phase 5).** Claude Code writes to `~/.claude/` for plugin state; a read-only root made the container unusable. Compensated with `--cap-drop=ALL`, `--no-new-privileges`, `chmod -R a-w` on the app payload, and UID 1001. Documented in `docs/superpowers/specs/2026-04-17-phase5-production-hardening-design.md`.
3. **Outer-loop-wrapper → `loop.stages:` (Phase F brainstorm).** The first cycle design wrapped the Archon run in a host-side shell `while` loop. User flagged that Archon couldn't detect a loop terminating from outside. Replaced with a declarative `loop.stages:` primitive inside the workflow YAML, with unrolled iterations as a fallback for fixed small counts.
4. **SSE "live dashboard" scope (Phase F F.3 dogfood).** We had assumed `workflows-run` would stream SSE events. Dogfood caught that `archon serve`'s SSE stream only observes runs launched *through the server* — CLI-launched runs write to SQLite but don't emit SSE. Corrected: `workflows-status` now honestly documents the split, and we pivoted the "live follow" story to (a) `archon`'s native stderr progress lines and (b) `docker logs -f` for per-container detail.

## Coupling risk — Archon

Three explicit coupling points documented by the solution-architect review (see `reviews/2026-04-19-phase-f-solution-architect.md`):

| Surface | What we read | Failure mode |
|---|---|---|
| SQLite schema | `remote_agent_workflow_runs`, `remote_agent_workflow_events` column names | Archon migration → our status skill breaks silently |
| Node primitive | `bash:` node type in workflow YAML | Archon drops/renames bash node → preprocessor output rejected |
| REST shape | `/api/workflows/runs` and `/api/workflows/runs/<id>` wrapper | Shape change → blank detail rows (we hit this in dogfood and fixed the unwrap) |

**Worst-case degradation is acceptable**: if Archon is abandoned, the generated bash nodes are plain `docker run` commands that still work by hand — the plugin degrades to a YAML templating layer rather than going dark. No pinned-version check yet; that's a v2 item.

## Uncaptured failure modes (surfaced by review, deferred)

- **Credential temp file leak under parallel early termination** — EXIT trap is workflow-scoped, not node-scoped; a host crash mid-workflow may leave the mode-600 temp file on disk.
- **Silent content clobber at fan-in** — two parallel nodes writing the same path merge via `git merge -X theirs` with no post-merge diff signal.
- **Orphan containers on loop node timeout** — bash `for` loop over `docker run` may not forward SIGTERM to the active child.

All three will be tracked in the F.7 follow-up issues.

## Dogfood lessons incorporated in this PR

- **`/sdlc-core:pr` skill** now explicitly requires re-running any integration suites whose counts the PR body cites, with an example and `gh pr edit` guidance for post-creation drift. Prevents the class of stale-test-claim this PR almost shipped with.
