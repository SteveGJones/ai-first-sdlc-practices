# sdlc-workflows v1 Scope Review — Day-1 UX

**Reviewer role:** UX/DX architect (CLI + skill-prompt + docs UX).
**Date:** 2026-04-19
**Sources:** quickstart.md, README.md, troubleshooting.md, the seven
SKILL.md files, CLAUDE-CONTEXT-workflows.md, and the scope brief.

**TL;DR.** This plugin is *powerful*. It is also *harder to adopt than
to skip*. A first-time SDLC practitioner has to absorb ~12 new concepts
and run two image builds before they get anything back — and the
"reward" at the end is a temp-dir full of commits they have to
cherry-pick. Against the baseline of "just use Archon + docker run +
your existing agents," the plugin's day-1 value is real but poorly
packaged. The fix is not more features; it's aggressive consolidation:
merge seven skills to four, pick one canonical doc, and make the first
run produce a visible win in one command rather than five.

---

## 1. First-touch audit — what the user sees

### Step A — install

From README.md §Quickstart:

```
/plugin install sdlc-workflows@ai-first-sdlc
/sdlc-workflows:workflows-setup
/sdlc-workflows:deploy-team dev-team
/sdlc-workflows:workflows-run sdlc-parallel-review
```

**Confusion point 1.** The README Quickstart says
`/sdlc-workflows:deploy-team dev-team` — but `dev-team` is not a team
that exists after `workflows-setup`. `workflows-setup` copies workflow
YAMLs and command prompts; it does *not* seed a `dev-team.yaml`
manifest. The user runs step 3, gets `manifest not found`, and has no
obvious next step from the README alone. Troubleshooting.md §"manifest
not found" points them at `/sdlc-workflows:manage-teams --create` —
but Quickstart.md (the "full walkthrough") tells them to hand-author
`my-team.yaml`. Three entry points, three different first teams.

**Confusion point 2.** README says step 4 runs `sdlc-parallel-review`.
That workflow is a *bundled* SDLC workflow — it references teams the
user hasn't created. Unless the bundled workflows ship as team-image-
agnostic (they don't: each node has `image: sdlc-worker:<team>`),
`workflows-run sdlc-parallel-review` will fail on node 1 with `image
not found` until the user builds every team image the workflow
references. The README does not warn about this.

### Step B — `/sdlc-workflows:workflows-setup`

From SKILL.md lines 36–278. What the user actually sees (quoted from
the skill's own emit blocks):

- If archon absent → `"Archon is not installed. Installing from
  https://archon.diy..."` then a `curl | bash`.
- After install → likely `"installed-not-on-path"` (macOS default).
  The skill emits an 11-line block instructing the user to add
  `~/.bun/bin` to PATH. **First instruction after install is a
  homework assignment.**
- Then `"Copied SDLC workflows: copied: …"` (4 workflow files).
- Then `"Copied SDLC commands: copied: …"` (~7 command files).
- Then `archon workflow list` — which now shows the bundled SDLC
  workflows *plus* Archon's own 13 bundled workflows. No disambiguation.
- Then a 3-5 minute `build-base.sh` + `build-full.sh` with no
  progress signal beyond Docker's own output.
- Final "8h. Summary report" with 7-line health check table.

**Confusion points:**

- The `installed-not-on-path` branch *is* handled well in the prompt
  text, but it sits at the wrong layer. The user just ran one
  command and is now being asked to edit `~/.zshrc`. On a skill
  that's supposed to be "first-time setup," this is a
  foot-shoot: we install something with known PATH breakage, then
  hand the user a remediation script. Either make the skill *offer
  to append the PATH export for them* (with confirmation), or
  install archon somewhere already on PATH.
- "Health check" (step 9) is a diagnostic and is very competently
  built, but it's gated behind a flag (`--health-check`). First-time
  users don't know they need it. Running it *unconditionally* at
  the end of the happy path would catch the PATH problem, the
  image-CLI problem, and the credential tier all in one pass.
- The skill emits an unqualified line: `"Ready for delegation."`
  This is the emotional peak. But the user now has Archon, two
  Docker images, workflow YAMLs, and no team. The next command
  (`workflows-run sdlc-parallel-review`) will fail. The "Ready"
  message is a lie.

### Step C — first workflow run

From workflows-run SKILL.md.

If the user follows the README, they run
`/sdlc-workflows:workflows-run sdlc-parallel-review`. The skill:

1. Verifies archon on PATH (dies with a PATH fix-up suggestion if not).
2. Verifies the workflow exists via `archon workflow list`. OK.
3. Detects `NEEDS_PREPROCESSING` (bundled workflows all use `image:`
   nodes). OK.
4. Resolves credentials. On macOS, Keychain tier works. OK.
5. `mktemp -d` a workspace, **rsync-copies the user's project into
   it** (with a 20-line exclude list), `git init` a fresh repo,
   commits a seed.
6. Preprocesses the workflow into `$WORKSPACE/.archon/workflows/
   .generated/…yaml`, copies it over the original.
7. Runs `archon workflow run <name> --no-worktree` inside the
   workspace with `CLAUDECODE`, `CLAUDE_CODE_SSE_PORT`, and
   `CLAUDE_CODE_IPC_FD` unset.
8. Emits a monitoring block listing four different monitoring
   surfaces.
9. When complete, prompts: "1. cherry-pick / 2. keep workspace /
   3. discard?" — and the user picks one of three post-run
   resolutions the docs never warned them would exist.

**Confusion points:**

- The `image: sdlc-worker:dev-team` in
  `sdlc-parallel-review.yaml` doesn't exist because the user hasn't
  built a dev-team image. Archon hits that node, `docker run sdlc-
  worker:dev-team` fails with `image not found`, and the run
  fails. Failure mode is late (only after preprocessing + workspace
  setup + archon spin-up). The skill does not pre-validate that
  every `image:` in the workflow resolves to a local Docker image
  before launching.
- The "three-way cherry-pick / keep / discard" prompt is the most
  surprising UX moment in the whole plugin. Nothing in the quickstart,
  README, or skill description prepares the user for "your workflow
  ran in a parallel git history and you now have to decide what to
  do with it." For an SDLC practitioner who expected "this edits my
  repo," this is going to feel wrong.
- Monitoring is not one surface — it's four
  (terminal stderr, `archon serve` UI, CLI snapshot, `docker logs`).
  The skill lists all four. The user has to pick one. Defaulting to
  *nothing* means the runs look silent.

**Net:** between `workflows-setup` and a successful first run, the
user typically needs to: edit `~/.zshrc`, hand-author a team manifest,
build a team image, understand the cherry-pick prompt, and know to
open a second terminal for `archon serve`. That's a lot of small cuts.

## 2. Mental model load

Concepts a user must grok before v1 makes sense, roughly in the order
they're hit:

1. **Archon** — a separate tool installed via `curl | bash` to
   `~/.bun/bin` (Bun runtime). Orchestrates DAGs.
2. **Archon CLI vs `archon serve`** — two different surfaces, only
   one of which shows certain runs.
3. **The `~/.bun/bin` PATH quirk** — not the plugin's fault but the
   user still has to absorb it.
4. **The three-tier Docker image model** — `base` / `full` / `team`,
   and the rule that `full` is "a source image, not runnable."
5. **Team manifests** — `.archon/teams/<team>.yaml` with
   `schema_version: "1.0"`, `plugins`, `agents`, `skills`, `context`
   fields. All four have their own sub-rules.
6. **Plugin IDs vs agent IDs** — `sdlc-core:verification-enforcer` is
   a plugin-qualified agent reference. The user is asked to author
   these from memory.
7. **Workflow YAML with `image:` nodes** — a *non-Archon-native*
   field that only our preprocessor understands.
8. **The preprocessor** — rewrites `image:` nodes to `bash: docker
   run`. Users don't invoke it directly, but troubleshooting
   references it by name, so they need the concept.
9. **Command prompt Markdown files in `.archon/commands/`** — what
   the node actually runs Claude on, separate from the workflow YAML.
10. **Credential tiers** — Keychain / volume / config.yaml, with
    `resolve_credentials.py --json` as the introspection tool.
11. **The `sdlc-claude-credentials` Docker volume** (Linux tier).
12. **The workspace temp-dir + seed-commit + cherry-pick model** —
    runs happen in an isolated git tree and you choose how to
    merge back.
13. **`--no-worktree`** — flag we always pass because containers
    provide isolation.
14. **`loop.stages:`** — our own extension on top of the preprocessor
    for cycles.
15. **SSE dashboard stream vs CLI-launched runs** — the SSE stream
    *does not see CLI-launched runs*. Users must know this to avoid
    debugging a non-bug.
16. **`workflows-status` REST-or-SQLite dual-source** — the skill
    silently falls back; the user sees a `Source: rest` or
    `Source: sqlite` banner and has to know what the difference means.
17. **Stale image detection** — teams-status flags
    `manifest updated > image built` as stale. Users must know to
    rebuild.
18. **The `CLAUDECODE` env-var unset** — not user-facing, but it's
    in the skill prompt and a reader of the prompt (which is the
    Claude operating the skill) needs to understand why.
19. **The chmod-lock on `~/.claude/agents/` inside the container**
    — users hit this if they try "add another agent" at runtime.
20. **`--direct-only` fallback mode** — buried in troubleshooting:
    a way to run the plugin without Archon at all. Users who need
    this need it *badly* and most will never find it.

### Brutally honest answer

**That list is roughly twice what a v1 should ask.** The essential
ones for "I want to run a workflow" are:

- Archon exists and orchestrates DAGs (1)
- Team manifests define what a container has (5)
- Workflow YAML defines the DAG (7)
- Command prompts say what each node does (9)
- Credentials come from my Keychain (10)

That's **five concepts**. Everything else is either:

- Plumbing that should be invisible (2, 3, 8, 11, 13, 15, 16, 18,
  19) — hide these behind the skills that use them.
- Advanced features the user does not need on day 1 (14, 17, 20) —
  defer to a "Day 2" page or cut entirely.
- Decisions we foisted on the user that we could have made for them
  (4, 12) — the three-tier image model and the cherry-pick workflow
  are our architecture; we should not make the user learn it just
  to delegate a review.

## 3. Per-skill UX verdict

### 3.1 `workflows-setup` — **KEEP, simplify**

- **User expectation:** "I just installed the plugin. Make it work."
- **Actual:** installs Archon (with PATH landmine), scaffolds dirs,
  copies bundled workflows/commands, builds two Docker images,
  emits a health-check summary.
- **Gap:** medium. The flow is correct but too much ceremony for
  something called "setup." The `--workflows-only` / `--with-docker`
  / `--health-check` flag matrix is a classic "features for
  features' sake."
- **Recommendation:**
  - Default is fine; cut `--workflows-only`. It's a footgun: the
    skill already documents it as "only use this when you know you
    already have the images." If you know that, you don't need a
    flag.
  - Fold `--health-check` into the end of the happy path. Always
    run it. No flag needed.
  - Offer to add `~/.bun/bin` to `~/.zshrc` after install (with
    confirmation). The 11-line "here's how to edit your shell
    init" block is a product failure dressed as documentation.
  - Seed a `general-purpose` team manifest by default. Today the
    skill scaffolds workflows but not teams, and the bundled
    workflows reference team images that don't exist. Either don't
    ship workflows that reference team images the user hasn't
    built, or build a default team for them.

### 3.2 `deploy-team` — **MERGE INTO manage-teams**

- **User expectation:** "Build the image for my team manifest."
- **Actual:** reads manifest, validates, builds CLAUDE.md,
  builds image, updates `image_built` timestamp.
- **Gap:** small — the skill does exactly what its name says.
- **But:** it's the tail end of the "create a team" flow. Every
  mode of `manage-teams --create` ends with *"Build the team image
  now? (a) Yes — run /sdlc-workflows:deploy-team"*. That's a two-
  command workflow with one natural flow.
- **Recommendation: merge.** `manage-teams --create` should end
  by *doing the build inline*, not handing off. Keep the build
  script (`build-team.sh`) as the engine. Lose the skill surface.
  If advanced users want raw build, document the shell one-liner:
  `bash plugins/sdlc-workflows/docker/build-team.sh <name>`.

### 3.3 `author-workflow` — **KEEP, trim**

- **User expectation:** "Generate a workflow for me."
- **Actual:** loads `CLAUDE-CONTEXT-workflows.md`, asks about
  teams + DAG, generates YAML + command prompts, validates, commits.
- **Gap:** small on the happy path. But the skill loads a 542-line
  context doc *before asking the first question*, and the "Designing
  long-running or cyclical workflows" section (lines 186-201) asks
  the user five follow-up questions about token cost and node
  duration on top of the main Q&A.
- **Recommendation:** keep. Trim step 0 to "read the schema section"
  not "read the whole 542-line context." And move the five
  cycle/cost questions to a separate `author-workflow --cycle` mode
  — the vast majority of workflows are linear.

### 3.4 `workflows-run` — **KEEP, but the cherry-pick prompt is the worst UX moment in v1**

- **User expectation:** "Run this workflow against my project."
- **Actual:**
  - rsync-copies your project into a tempdir (with a 20-line
    security exclude list)
  - git inits a seed commit there
  - runs archon inside the tempdir
  - on completion, prompts you to cherry-pick / keep / discard
  - cherry-pick is manually implemented via
    `git format-patch | git apply --index` because the commit
    objects live in a separate `.git` dir
- **Gap:** large. Users expect "this edits my repo" or "this edits
  a branch." They get "your work lives in /tmp/sdlc-run-XXXXXX and
  now you have to choose." The reasons are all defensible (security
  excludes, no parent-repo hook contamination, no accidental push),
  but they are not obvious from the skill description.
- **Recommendation: keep the skill, rework the default.**
  - **Default to option 1 (cherry-pick).** Today the prompt
    defaults to 1 "when the user is absent or replies unclearly."
    Make that the *first-class* path and don't ask unless a flag
    `--keep-workspace` or `--discard` was passed.
  - Show the diff *summary* (files changed, net lines) before
    cherry-pick, not a question. Users want to see outcomes, not
    adjudicate them.
  - Document the "your work ran in a parallel git history" reality
    in the quickstart, *before* the user hits it. The current
    quickstart says "use the skill: /sdlc-workflows:workflows-run
    my-pipeline" and jumps straight to monitoring. The three-way
    prompt appears nowhere in the quickstart.
  - Pre-flight validation: before we rsync and git-init, check
    that every `image:` referenced by the workflow is a local
    Docker image. This is a one-line docker inspect loop and it
    saves the user 20+ seconds of setup before a 1-line failure.

### 3.5 `workflows-status` — **KEEP**

- **User expectation:** "What ran, what's running, what failed."
- **Actual:** REST-first, SQLite-fallback query helper. Supports
  `--running`, `--recent N`, `<run-id>`.
- **Gap:** small. Honest about the SSE-vs-CLI-launched-runs
  asymmetry (good), and the fallback is transparent.
- **One trim:** the skill doc is 95 lines; the companion skills
  list (§3 "Companion skills") belongs in a single index, not at
  the bottom of every status skill.

### 3.6 `manage-teams` — **KEEP, but **this is the real "setup" skill for teams** and should own the team-create flow**

- **User expectation:** "Help me create/update/delete/inspect
  teams."
- **Actual:** five modes — `--create`, `--update`, `--delete`,
  `--review`, `--plan-task` — each a Q&A flow. The coaching
  framing is genuinely nice.
- **Gap:** small on individual modes, but the five-mode surface
  and the handoffs (to `deploy-team`, to `author-workflow`) make
  it feel like a switchboard. The `--plan-task` mode in particular
  is *workflow design* — it overlaps with `author-workflow`.
- **Recommendation:**
  - Keep `--create`, `--update`, `--delete`, `--review`. These
    are genuine team-lifecycle actions.
  - Absorb `deploy-team` as the default tail of `--create` and
    `--update` (see 3.2).
  - **Cut `--plan-task`**, or merge it with `author-workflow`.
    Two skills asking "what's the task?" and recommending a DAG
    is two steps when it should be one. Pick one home for task-
    to-workflow recommendation.

### 3.7 `teams-status` — **MERGE INTO manage-teams --review**

- **User expectation:** "Show me what teams I have."
- **Actual:** fleet table + staleness + workflow coverage +
  coaching signals + plugin-environment changes.
- **Gap:** the output is great. The problem is that
  `manage-teams --review` already runs
  `teams_status_report.py` and `coaching_signals.py` (the exact
  same scripts) and presents "Delegation workforce review: …"
  which is the same table in prose form.
- **Recommendation:** merge. `/sdlc-workflows:teams-status` →
  keep as a shortcut alias if you like, but all the intelligence
  should live in `manage-teams --review`. Two skills that wrap
  the same two Python scripts with different wrapper text is
  exactly the duplication a v1 should cut.

### Summary: 7 skills → 4

| Current | Proposed |
|---|---|
| workflows-setup | **workflows-setup** (simpler, installs default team) |
| deploy-team | *(merged into manage-teams)* |
| author-workflow | **author-workflow** (trimmed) |
| workflows-run | **workflows-run** (pre-flight + default cherry-pick) |
| workflows-status | **workflows-status** (unchanged) |
| manage-teams | **manage-teams** (absorbs deploy-team + teams-status) |
| teams-status | *(merged into manage-teams --review)* |

## 4. Friction hotspots (ranked — highest give-up risk first)

1. **The `~/.bun/bin` PATH dance after `workflows-setup`.** The
   skill does the right thing by distinguishing `installed`/
   `installed-not-on-path`/`missing`, but the user's first
   post-install action is still "edit ~/.zshrc" or "remember to
   export PATH in every new shell." First-run friction is
   disproportionate; this one bites *every* macOS user once.
   *Citation:* workflows-setup/SKILL.md §2.

2. **`deploy-team dev-team` in the README with no `dev-team.yaml` to
   deploy.** The public Quickstart in README.md explicitly tells
   users to run `deploy-team dev-team` immediately after setup. No
   such manifest exists. Users will Google "sdlc-workflows manifest
   not found" before finding `manage-teams --create`.
   *Citation:* README.md lines 55-60.

3. **`workflows-run sdlc-parallel-review` fails at node 1** with no
   pre-flight image check. The bundled workflow references team
   images that only exist if the user has deployed those teams.
   Failure mode: archon starts, preprocessing runs, workspace is
   prepared, *then* docker run fails. Feels slow and mysterious.
   *Citation:* workflows-run/SKILL.md §2b (no image-existence check
   before launch).

4. **The cherry-pick prompt at end of run.** User expected their
   project to have new commits. Instead they get a three-option
   prompt referencing a tempdir. Even the best-case (option 1,
   cherry-pick) requires reading the prompt carefully.
   *Citation:* workflows-run/SKILL.md §6.

5. **Four monitoring surfaces listed simultaneously** with no
   default. The skill emits "Live per-node lines in this
   terminal / Archon web UI / CLI snapshot / Container-level
   detail" — all correct, all useful, none obviously the right
   first choice. The quickstart duplicates this list at step 4.5.
   A user wanting to "watch the run" sees four commands they don't
   know and gives up on watching.
   *Citation:* workflows-run/SKILL.md §2b "While it runs" block;
   quickstart.md §"Step 4.5: Watch It Run".

## 5. Doc-to-reality drift

### Quickstart vs code

1. **Quickstart §Step 4 inline-CLI fallback writes to
   `.archon/workflows/.generated/` and `archon workflow run` from
   the project root.** The actual `workflows-run` skill does
   neither — it `mktemp`s a workspace, rsyncs in, and runs from
   there. A user who "just uses the CLI directly per the quickstart"
   gets a fundamentally different execution shape than a user who
   uses the skill: no workspace isolation, no seed commit, no
   cherry-pick step. If they compare outputs across approaches
   they'll be confused about where their commits went.
   *Citation:* quickstart.md lines 141-162 vs workflows-run/SKILL.md
   §2b.

2. **Quickstart step 5 says "git log --oneline -5 — See commits from
   each container"** as if commits land in the *user's* repo. In
   the skill path, they land in the tempdir until the user picks
   cherry-pick. The quickstart does not mention the prompt at all.
   *Citation:* quickstart.md lines 214-219.

3. **README Quickstart uses `deploy-team dev-team`** (expects
   manifest to already exist), while `quickstart.md` Step 2 says
   "Create `.archon/teams/my-team.yaml`" by hand. Neither references
   `manage-teams --create`, which is the *actual* scaffolder.
   Three-way drift: README, quickstart.md, manage-teams skill.

4. **Quickstart does not warn about CLAUDE_TIMEOUT=300**, but
   `author-workflow` SKILL.md §"Designing long-running…" (and
   troubleshooting.md) make clear that 300s is too short for real
   work. A first workflow that does anything substantial hits
   timeout with no forewarning from the quickstart.

5. **Quickstart §"Or Use the Authoring Skill"** says
   `/sdlc-workflows:author-workflow` "asks what you want to build,
   generates the workflow YAML, command prompts, and validates
   everything." True, but it also *commits* on the user's behalf
   (skill §7). That's a surprise. Silent `git commit` inside a
   skill is the kind of thing that breaks trust on day 1.
   *Citation:* author-workflow/SKILL.md lines 166-170.

### Code vs docs (code does more than docs say)

6. **`workflows-run` rsync-excludes 20 paths** including SSH,
   AWS, GCP, kube, npm, Docker config, shell history. Users are
   never told this. A user who *expected* their `.env` to be
   available in the container (reasonable assumption for many SDLC
   tasks) will spend time debugging "why can't my container see
   my project config?" Document this.
   *Citation:* workflows-run/SKILL.md §2b rsync block.

7. **`workflows-run` unsets `CLAUDECODE`, `CLAUDE_CODE_SSE_PORT`,
   `CLAUDE_CODE_IPC_FD` before calling archon** because of an
   Archon 1.x warning path. The user is running this skill inside
   Claude Code, where those vars are set. This is not a user-
   facing issue *unless* the workaround ever stops working, and
   then the symptom ("workflow discovery silently fails") is very
   confusing. Worth a troubleshooting entry.
   *Citation:* workflows-run/SKILL.md lines 197-200.

## 6. Recommendations

### 6.1 Skills to cut / merge

| Skill | Verdict | Rationale |
|---|---|---|
| `deploy-team` | **CUT (as a separate skill surface)** | Its only caller is the tail of `manage-teams --create` / `--update`. Keep the `build-team.sh` engine; drop the skill wrapper. |
| `teams-status` | **CUT (merge into `manage-teams --review`)** | Same scripts, same output, different wrapper text. Two skills for one job. |
| `manage-teams --plan-task` | **CUT or MERGE with `author-workflow`** | "What task do you want to do, and what workflow/teams fit?" is also exactly what `author-workflow` asks. Pick one home. Recommend merging into `author-workflow` (since its output is a workflow file anyway). |
| `workflows-setup --workflows-only` flag | **CUT** | Footgun by its own admission; no valid first-run use. |
| `workflows-setup --health-check` flag | **FOLD IN** | Always run at end of setup. No flag. |

Net: seven skills become **four** — `workflows-setup`,
`manage-teams`, `author-workflow`, `workflows-run`,
`workflows-status`. That's five, and four is aspirational; if
`workflows-status` remains a separate skill because it's stateless
and useful in isolation, keep it.

### 6.2 Docs to consolidate

Today we have:

- README.md (100 lines) — install + quickstart + troubleshooting
  pointer + Apple Silicon note.
- quickstart.md (266 lines) — full walkthrough.
- troubleshooting.md (281 lines) — symptom → fix.
- CLAUDE-CONTEXT-workflows.md (542 lines, repo root) — canonical
  schema reference.
- 7 × SKILL.md — each with its own inline "companion skills" and
  monitoring blocks.

**Recommended consolidation:**

- **Keep CLAUDE-CONTEXT-workflows.md.** This is load-on-demand
  reference for Claude. It's well-scoped.
- **Merge README.md + quickstart.md → one `README.md`.** Today
  the README's "Quickstart" block and quickstart.md's "Step 1-5"
  disagree on whether to use `deploy-team` or hand-author
  `my-team.yaml`. One file, one story, one correct sequence.
- **Keep troubleshooting.md** as a flat symptom index. Cut the
  duplicated per-skill "companion skills" footers and point them
  here.
- **Cut the "Long-Running Workflows, Cycles, and Monitoring"
  section out of CLAUDE-CONTEXT-workflows.md** (and out of
  author-workflow SKILL.md) into a dedicated
  `docs/cycles-and-monitoring.md` loaded only when the user's
  workflow actually uses `loop.stages:` or runs > 5 min. Day 1
  doesn't need it; day 7 does.

### 6.3 First-run flow rewrite

The v1 day-1 path should be **four commands**, end to end:

```
/plugin install sdlc-workflows@ai-first-sdlc
/sdlc-workflows:workflows-setup
/sdlc-workflows:workflows-run sdlc-code-review
# (see what it did)
```

For this to work:

1. `workflows-setup` must install a default `general-purpose`
   team and build its image. Not optional. The user did not ask
   for a team; they asked for workflows. Make workflows work.
2. Bundled workflows must either reference only the default team,
   or ship as "commissioned" stubs that `manage-teams --plan-task`
   fills in per project. Today they reference team images that
   don't exist.
3. `workflows-run` must pre-flight-check that every `image:` in
   the workflow resolves before launching. Fail fast, with a
   one-line remediation:
   `"Workflow references sdlc-worker:dev-team which is not
   built. Run: /sdlc-workflows:manage-teams --create"`.
4. Cherry-pick must be the *default action*, not a prompt. If
   we want the user to inspect first, show the diff summary and
   commit automatically, then say
   *"Cherry-picked 3 commits onto main. Workspace kept at <path>
   if you want to re-run. Use --discard next time to clean up."*

## 7. Is it easier to install-and-use this than to roll your own?

**Honest answer:** not yet — but the gap is small and closable.

"Rolling your own" baseline: install archon, write a workflow YAML,
`docker run` each node against your existing claude-code install. For
a small one-off job, that's ~an hour and gives you something working.

sdlc-workflows baseline today: install archon, build 2 images, author
a manifest, build a 3rd image, hand-author or generate a workflow,
learn the cherry-pick prompt, pick a monitoring surface. For a
first-time user, that's **half a day** of reading and poking.

The plugin's real wins over the baseline — three-tier images,
credential tiers, coaching skills, workspace isolation, resumable
runs, stale-image detection, `loop.stages:` — are all genuine. None
of them land on day 1 in the current shape, because the user isn't
past the onboarding wall yet.

**If the four-command happy path above works, the plugin wins
decisively.** A user who types four commands and sees commits land
from a delegated review will stay. A user who spends 90 minutes
editing YAML and debugging PATH issues before their first successful
run will churn. The day-1 experience, not the feature list, is the
product.
