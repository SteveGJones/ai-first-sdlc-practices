# Technical Debt

Technical debt is the implied cost of additional rework caused by choosing an easy short-term solution instead of a better approach that would take longer. The metaphor was coined by Ward Cunningham.

Like financial debt, technical debt accrues interest in the form of slower future development. Teams can choose to take on debt deliberately to ship faster, but they should track it and plan to repay it.

This framework enforces a zero technical debt policy for application code. There must be no commented-out code, no placeholder TODO markers, and no unjustified suppression of warnings.
