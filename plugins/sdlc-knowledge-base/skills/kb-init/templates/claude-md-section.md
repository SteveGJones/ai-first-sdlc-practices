## Knowledge Base

This project uses the `sdlc-knowledge-base` plugin to maintain a structured, citable knowledge base. Decisions can be grounded in evidence rather than opinion. The library is updated by agents and queried by humans and agents alike.

### Layout

- **Library files** live in `library/` (one markdown file per topic)
- **Raw sources** live in `library/raw/` (immutable; agents read but never modify)
- **Shelf-index** lives in `library/_shelf-index.md` (machine-readable, hash-tracked, regenerated by `kb-rebuild-indexes`)
- **Log** lives in `library/log.md` (chronological append-only record of ingest/query/lint events)

### Library file format

Every library file has YAML frontmatter and a fixed section structure.

```markdown
---
title: "Topic name"
domain: comma, separated, topic, tags
status: active
tags: [array, of, search, terms]
source: library/raw/path-to-raw-result.md
cross_references:
  - library/related-file-1.md
  - library/related-file-2.md
---

## Key Question
The research question this file answers — one sentence.

## Core Findings
1. Specific claim with specific number — citation (author, year, sample size or methodology)
2. Specific claim with specific number — citation
3. ...

## Frameworks Reviewed
| Framework | Evidence base | Limitations |
|---|---|---|
| ... | ... | ... |

## Actionable Thresholds
| Metric | Threshold | Source | Signal |
|---|---|---|---|
| ... | ... | ... | ... |

## Design Principles
What the research says about how to build, not just what to measure.

## Key References
1. Full citation (author, year, title, venue)
2. ...

## Programme Relevance
How this maps to specific project artifacts, issues, and decisions.
This section is project-specific and the most maintenance-intensive.
```

Required frontmatter: `title`, `domain`, `status`, `source`. Optional: `tags`, `cross_references`, plus any project-specific extensions.

### The three operations

The plugin provides three operation skills following Karpathy's framing:

| Operation | Skill | Agent invoked | Purpose |
|---|---|---|---|
| **Ingest** | `/sdlc-knowledge-base:kb-ingest <source>` | `agent-knowledge-updater` | Process a new source, update existing files where it touches them, create new files where needed, update the shelf-index, append to log.md |
| **Query** | `/sdlc-knowledge-base:kb-query <question>` | `research-librarian` | Answer a question against the library with structured evidence and citations |
| **Lint** | `/sdlc-knowledge-base:kb-lint` | `research-librarian` (lint mode) | Health-check: contradictions, orphans, missing cross-references, stale claims, data gaps |

Plus three management skills:

| Skill | Purpose |
|---|---|
| `/sdlc-knowledge-base:kb-rebuild-indexes` | Hash-based incremental rebuild of the shelf-index |
| `/sdlc-knowledge-base:kb-promote-answer-to-library` | File a query result back into the library as a new page (Karpathy's "answers compound" insight) |
| `/sdlc-knowledge-base:kb-validate-citations` | Spot obvious citation hallucinations (DOI resolution, arXiv ID validation) |

### Agents and models

| Agent | Default model | Why |
|---|---|---|
| `research-librarian` | `sonnet` | Needs reliable command-line tool use (`find`, `grep`) for filesystem navigation. Haiku struggles with these patterns. |
| `agent-knowledge-updater` | `sonnet` | Same reasoning. Plus the ingest workflow involves judgment about what to update vs create. |

To override (e.g., use Haiku for cost reasons on a project with limited filesystem navigation needs), set the model in the agent file's frontmatter.

### Anti-hallucination rules

These are non-negotiable for the librarian. The trust users place in the knowledge base depends on them:

1. **Never invent a citation.** Findings without citations in the library file are not reported.
2. **Never invent a statistic.** Numbers not in the library file are not produced.
3. **Never paraphrase a topic the library doesn't cover.** When asked about something outside the library, the response is: "The library has no evidence on this." Then optionally suggest the closest related entries or the next step (`kb-ingest` for new sources, external research).

### Workflow

**To add a source to the library:**
```
/sdlc-knowledge-base:kb-ingest path/to/raw/source.md
```
Or for a URL:
```
/sdlc-knowledge-base:kb-ingest https://example.com/article
```

The `agent-knowledge-updater` reads the source, identifies which existing library files it touches, updates them, creates new files where needed, updates the shelf-index, and appends to the log.

**To query the library:**
```
/sdlc-knowledge-base:kb-query "What does our research say about X?"
```

The `research-librarian` reads the shelf-index, identifies the 2-4 most relevant library files, deep-reads only those, and returns structured evidence with citations.

If the answer is worth keeping for future queries, promote it:
```
/sdlc-knowledge-base:kb-query "..." --promote-to-library
```

**To health-check the library:**
```
/sdlc-knowledge-base:kb-lint
```

The lint operation checks for contradictions between files, stale claims newer sources have superseded, orphan files with no inbound cross-references, important concepts mentioned but lacking their own page, missing cross-references, and data gaps. Returns a report; does not auto-fix.

**After editing library files manually:**
```
/sdlc-knowledge-base:kb-rebuild-indexes
```

Hash-based incremental rebuild. Only re-extracts the changed entries.

### Optional: environment validation integration

To have environment validation check for stale shelf-index entries automatically, opt in by setting:

```
knowledge_base.staleness_check.enabled: true
knowledge_base.staleness_check.mode: warning  # or "strict" for hard failure
```

When enabled, `validate --pre-push` runs `kb-rebuild-indexes` in dry-run mode and reports any files where the hash has drifted from the recorded value. In `warning` mode, this is informational. In `strict` mode, validation fails until the index is rebuilt.

Default: opt-in is OFF. Installing the plugin does not change validation behaviour.

### External libraries (cross-library query)

This project can query one or more external knowledge bases alongside its own. External libraries ("corporate asset libraries") are registered once in the user-scope registry and activated per-project.

**To register an external library** (once, per library per machine):

```
/sdlc-knowledge-base:kb-register-library <handle> <absolute-path-to-library-dir> [description]
```

This appends an entry to `~/.sdlc/global-libraries.json`. Hand-editing that file is also supported — see the plugin's README.

**To activate a registered library in this project**, add its handle to `.sdlc/libraries.json`:

```json
{
  "version": 1,
  "activated_sources": ["corporate-semiconductor"]
}
```

When at least one external library is activated, every `kb-query` invocation dispatches the librarian against each activated source in parallel. Results are attributed (`[local]`, `[corporate-semiconductor]`, etc.) and the local library's context primes the external queries.

**What lives where:**

- Handle + filesystem path + description: `~/.sdlc/global-libraries.json` (user-scope — never committed to this repo)
- Activation list (handles only): `.sdlc/libraries.json` (project-scope — safe to commit)

**Confidentiality:** hard isolation between concurrent confidential engagements is not enforced by the plugin. If two engagements must not share findings, keep them in separate repositories. Each repo's `.sdlc/libraries.json` activates only the appropriate handles.

### When NOT to use the knowledge base

- For decisions where evidence isn't needed (routine maintenance, simple bug fixes)
- For ephemeral information that doesn't need to compound over time
- For project context that auto-memory already captures (user preferences, session state) — those belong in the auto-memory system, not the knowledge base

The knowledge base is for **domain evidence about a problem space**: research, frameworks, thresholds, citable findings. Other knowledge stores serve other purposes.
