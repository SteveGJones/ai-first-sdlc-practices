## Critical implementer guidance

### Authoring REQ statements

Each REQ statement is one declarative sentence in the form *"The [component] MUST [observable behaviour]."* Examples that pass the dogfood-signal bar:

- **Good:** *"The `id_uniqueness` validator MUST emit one error per duplicated ID, citing every source file containing that ID."*
- **Good:** *"The `commission-assured` skill MUST scaffold an empty `docs/change-impacts/` directory regardless of whether change-impact-gate is enabled at commissioning time."*
- **Bad:** *"The `id_uniqueness` validator exists in `traceability_validators.py`."* (paraphrased file-presence)
- **Bad:** *"The `id_uniqueness` validator works."* (untestable)

If a REQ feels like it's just describing the function signature, stop and either (a) elevate to a user-visible capability or (b) drop it and log a finding: *"Component X has no user-visible capability worth specifying — Method 2 doesn't gracefully handle pure-utility code."*

### Authoring DES statements

DES describes the chosen implementation approach at one level deeper than REQ. Cite via `**satisfies:** REQ-<feature>-NNN`. One DES per REQ minimum; more if the implementation has multiple architectural choices worth recording.

### Authoring TEST statements

TEST describes the verification approach. Cite both REQ and DES via `**satisfies:** REQ-<feature>-NNN via DES-<feature>-NNN`. The existing `tests/test_*.py` files are the implementation of the test-spec; the test-spec describes intent.

### Annotation rule

For Python files: annotate every PUBLIC function (no leading underscore) with `# implements: <DES-ID>` as the first line of the function body. Skip private helpers (`_module_paths`, `_flush`, `_index_by_satisfies`, etc.).

For markdown files (SKILL.md, agent files): use HTML-comment form `<!-- implements: <DES-ID> -->` immediately after the frontmatter or top-level heading. **The existing `_IMPLEMENTS_RE` regex will not match this form** — log F-001 as a finding when this is first encountered (Task 5).

### Findings file format

Append to `research/phase-f-dogfood-findings.md`. Each finding:

```markdown
### F-NNN — <Short title>

**Severity:** BLOCKER / IMPORTANT / MINOR
**Surfaced during:** Task N (Stage X)
**Reproduction:** <how to reproduce in 2-3 lines>
**Impact on regulated-industry users:** high / medium / low / none
**Suggested resolution:** fix in v0.2.0 / defer / clarify in docs / no action
**Related code/skill/validator:** <pointer>
```

Number findings sequentially as F-001, F-002, etc.

### What "test-fail-then-pass" looks like in this phase

Most tasks don't have a TDD code cycle. Instead each artefact-authoring task uses this rhythm:

1. Run `cited_ids_resolve` against the partial registry → observe it failing because the new REQs aren't cited yet (or DES/TEST aren't authored yet)
2. Author the artefacts
3. Re-run validator → observe pass
4. Commit

For tasks that ONLY author REQs (no DES/TEST yet), the validator pass criterion is just *"the new REQs parse and appear in `library/_ids.md` after rebuilding it"*. Skip validation gates that aren't yet meaningful.

```markdown
## Motivation

Method 2 (Assured) needs to query across spec types (REQ, DES, TEST, CODE) when answering "what implements X?" queries. This requires extending the existing synthesis-librarian agent with a new dispatch mode. The mode lives in the kb-bridge module — the bridge between the Assured bundle and the existing knowledge-base agent infrastructure.

## Requirements

### REQ-kb-bridge-mode-001

The synthesis-librarian agent MUST recognise a `mode: synthesise-across-spec-types` dispatch directive and treat it as a distinct synthesis mode, not a fallback to `SYNTHESISE-ACROSS-SOURCES`.

**Module:** P1.SP1.M3

### REQ-kb-bridge-mode-002

When the SYNTHESISE-ACROSS-SPEC-TYPES mode is active, the agent's `valid_handles` whitelist MUST accept the four pseudo-handles `req`, `des`, `test`, `code` in addition to any project library handles, so that spec-type-keyed dispatch sources resolve correctly during attribution post-checks.

**Module:** P1.SP1.M3

## Out of scope

- Multi-mode dispatch (one mode per dispatch is enforced by the existing dispatcher).
- New synthesis prompts for non-Assured bundles.

## Success criteria

- A dispatch with `mode: synthesise-across-spec-types` and pseudo-handles `req`, `des`, `test`, `code` produces an output with `## Synthesis` and `## Attribution` sections, citing IDs (not file paths).
- The same dispatch with one of the four pseudo-handles missing from `valid_handles` aborts via the post-check (per Article confidentiality).
```

```markdown
# Phase F — Dogfood Findings

## F-001 — `# implements:` annotation parser does not handle markdown files

**Severity:** IMPORTANT
**Surfaced during:** Task 4 (Stage 1 proof-of-flow)
**Reproduction:** The `_IMPLEMENTS_RE` regex in `plugins/sdlc-assured/scripts/assured/code_index.py` matches `^\s*#\s*implements:\s*(?P<ids>.+)$`. In a markdown file, the line `# implements: REQ-x-001` parses as an H1 heading by markdown renderers AND matches the regex — producing a fake annotation that displays as a giant title. The HTML-comment workaround `<!-- implements: REQ-x-001 -->` displays correctly in markdown but does NOT match the regex.
**Impact on regulated-industry users:** high — most user-facing artefacts (SKILLs, agents, design docs) are markdown, not Python. Regulated-industry users will hit this within the first feature.
**Suggested resolution:** v0.2.0 — extend `_IMPLEMENTS_RE` to recognise both forms, or add a second regex `_HTML_IMPLEMENTS_RE = re.compile(r"<!--\s*implements:\s*(?P<ids>.+)\s*-->")`. Alternative: define a project-wide rule that markdown annotations live in YAML frontmatter (`implements: [DES-x-001, ...]`).
**Related code:** plugins/sdlc-assured/scripts/assured/code_index.py:9-13, plugins/sdlc-assured/scripts/assured/traceability_validators.py:_IMPLEMENTS_RE
```
