# Design Spec: Programme + Assured Bundles (Joint Delivery)

**Date:** 2026-04-26
**EPIC:** #178 (subsumes #103 + #104; absorbs #98 as Phase C)
**Branch:** `feature/sdlc-programme-assured-bundles`
**Status:** Phase A complete (this document); Phase B-G to follow

## 1. Executive summary

The framework gains two new commissionable SDLC bundles: **Programme** (waterfall phase gates with cross-phase review enforcement) and **Assured** (everything in Programme plus stable identifiers, bidirectional traceability, declarative decomposition, KB-for-code annotation indexing, and audit-friendly export pipelines). They ship together because Method 2 (Assured) is structurally Method 1 (Programme) with a richer underlying data model — building Method 1 alone is the half-step.

The bundles install via `/sdlc-core:commission` (Phase C, absorbing existing sub-feature #98). Both are dogfooded on this repo before external recommendation (Phase F). The full set ships at PR time on a single long-lived branch following the EPIC #164 / PR #177 precedent.

This spec is the input to plan-writing (`superpowers:writing-plans`) for Phases C-F. Phase B (decomposition spike) runs in parallel, validating that Method 2's decomposition primitive is workable on this repo before validators are built around it.

## 2. The two methods at a glance

(Drawn from METHODS.md Section 2, with Phase A updates.)

| Bundle | Method | Purpose | Substrate shared with the other |
|---|---|---|---|
| **Programme** | Method 1 — formal waterfall phase gates | Programme-of-work scale; explicit phase boundaries; cross-phase review enforcement | Templates, gate validators, `phase-review` skill, `traceability-export` (basic) |
| **Assured** | Method 2 — agent-first specs with traceability + decomposition + KB-for-code | Regulated industries; defensible bidirectional traceability across DO-178C / IEC 62304 / ISO 26262 / IEC 61508 / FDA 21 CFR Part 820 simultaneously | Same Programme substrate, plus ID registry, decomposition validators, `kb-codeindex`, librarian extensions, change-impact pattern, render pipeline, standard-specific export formats |

Method 2 is a strict superset of Method 1 in skills and validators. The two bundles are independently installable; commissioning to Assured implies commissioning to Programme.

## 3. Method 1 (Programme bundle) detail

(METHODS.md Section 3; Phase A scope changes 1-4 incorporated.)

### Phase artefacts

Per feature, four artefacts in `docs/specs/<feature>/`:

- `requirements-spec.md` — what the feature must do
- `design-spec.md` — how it will be built, citing the requirements-spec
- `test-spec.md` — what tests verify, citing requirements-spec and design-spec
- (Code, written under TDD discipline driven by the test-spec)

### Phase gates

Validators that block progression unless gate condition is met:

- `requirements-gate`, `design-gate`, `test-gate`, `code-gate`
- **Block on missing artefact AND on broken cross-phase references.** Broken references are defects, not style issues. *[Phase A change 1; cited from Line 1 §3 Claims 1, 5; Line 3 §3 Claims 1-3]*
- Warn on weak (vague but non-broken) references.

### Skills shipped (Programme bundle)

- `commission-programme` — installs the bundle
- `phase-init <phase>` — instantiates phase artefact from template
- `phase-gate <phase>` — runs the gate validator
- `phase-review <phase>` — **mandatory** for design-spec and test-spec phases. Mandatory because syntactic gate ≠ semantic soundness. *[Phase A change 4; cross-line agreement 1]*
- `traceability-export <format>` — produces audit-friendly matrices in basic formats (CSV table, markdown matrix). *[Phase A change 2; Line 1 §3 Claim 6]*

### Model selection hints (per skill)

Each skill states *what kind of work it is*; operator chooses model accordingly. *[Phase A change 3; cross-line agreement 1]*

- `phase-review` — structured comparison and reasoning task → reasoning-strong model
- `phase-gate` — mechanical syntax-checking → smaller/faster model fine
- `phase-init` — template instantiation → smaller models sufficient
- `commission-programme` — one-shot configuration → default model

### Integration with existing superpowers

- `brainstorming` → produces requirements-spec
- `writing-plans` → produces design-spec
- TDD → drives test-spec + code

Composition, not replacement.

## 4. Method 2 (Assured bundle) detail

(METHODS.md Section 4; Phase A scope changes 1-10 incorporated.)

### Identifier system

Stable IDs across spec types: `REQ-<feature>-<num>`, `DES-`, `TEST-`, `CODE-`. For decomposed systems: `<program>.<sub-program>.<module>.<type>-<num>` (e.g. `P1.SP2.M3.REQ-007`). Default decomposition is `P1.SP1.M1.*` for any project that hasn't declared structure.

### ID registry

`library/_ids.md` auto-generated by `kb-rebuild-indexes`. Tracks every ID, source artefact, and links.

Validators: ID uniqueness; cited IDs exist in registry; no orphan IDs.

#### Decomposition refactoring without ID loss

When teams refactor decomposition (merging programs, moving modules), `kb-rebuild-indexes` remaps old IDs to new paths, preserving historical traceability. Mitigates premature-decomposition risk. *[Phase A Method 2 change 2; Line 2 §3 Claim 6]*

### Spec-as-KB-finding

Each REQ/DES/TEST/CODE record is structurally a KB finding (title, terms, facts, links). Reuses existing shelf-index entry shape. Existing `research-librarian` queries against this without modification. Existing `synthesis-librarian` extends to a new mode `SYNTHESISE-ACROSS-SPEC-TYPES`; `valid_handles` whitelist extends naturally to spec-type pseudo-handles.

### Bidirectional traceability

Forward links declared in linking artefacts. Backward indices regenerated by `kb-rebuild-indexes`.

Validators (Phase A change 3 incorporated):

- **Forward link integrity** — cited IDs exist; broken references block commit
- **Backward coverage** — every REQ has DES; every DES has TEST; every TEST has CODE annotation; every CODE has REQ/DES annotation. Orphans block.
- **Index regenerability** — `kb-rebuild-indexes` is idempotent (byte-identical output on repeat runs). Auditors verify by running and comparing. Warn on stale indices (default 7 days). *[Cross-line agreement 8; Line 1 §3 Claim 4]*
- **Annotation format integrity** — `implements: <ID>` references resolve; broken/malformed annotations block push. *[Line 2 §3 Claim 5]*

#### Optional validators (configured at commissioning)

- **`change-impact-gate`** (default disabled) — blocks commits where code changes lack change-impact annotation. Recommended for IEC 62304 / FDA 21 CFR Part 820 / ISO 26262 ASIL C/D. Commissioning script enables when regulatory context warrants. *[Phase A change 5; cross-line agreement 6; Line 1 §3 Claim 5]*

### Decomposition

(Phase A change 1 — full enrichment.)

Declarative, rooted in **DDD bounded contexts** layered with **Bazel visibility-rule discipline**. A module is a bounded context with consistent ubiquitous language, owned by one team or part of one team. Modules group into sub-programs; sub-programs into programs. *[Line 2 §3 Claims 1, 2, 4; cross-line agreement 5]*

The `programs` block at commissioning declares:

- Which programs exist; sub-programs each contains; modules each sub-program contains
- Directory paths per module
- **Granularity per module**: `granularity: [requirement | function | module]`. Default `requirement`. *[Phase A change 7; cross-line agreement 7]*
- **Visibility rules**: which modules may depend on which (context-map document). *[Phase A change 1]*
- **Optional within-module structure**: hexagonal architecture (`structure: hexagonal`) with `core` / `adapter` directory tagging. *[Phase A change 4; Line 2 §3 Claim 3]*

Validators (in addition to ID-registry validators):

- Every REQ has module assignment
- Every CODE annotation maps to declared module path
- Visibility-rule enforcement (imports respect declared visibility; circular cross-program deps block)
- **Anaemic-context detection** — code implementing REQ/DES not co-located in declared module flagged as boundary violation. *[Line 2 §3 Claim 4]*
- **Granularity match** — actual annotations match declared granularity (warn on under-specification)

### Organisational metadata (optional)

Owner / reviewer / approver are optional in spec YAML frontmatter, never schema-baked. Process enforcement lives in Git hooks / CI / workflow tools, not the spec schema. Prevents framework breaking when organisations reorganise. *[Phase A change 10; Line 3 §4 Rejection C]*

### KB extension to code (annotation-driven)

Inline comment annotations link CODE → DES / REQ:

```python
def authenticate(token: str) -> AuthResult:
    # implements: DES-auth-005, REQ-auth-003
    ...
```

`kb-codeindex` parses annotations, emits `library/_code-index.md` (shelf-index-shaped). Librarian queries it like the regular shelf-index. Output `[code]`-tagged with file:line locations.

**Explicitly NOT**: AST parsing, semantic search, call graphs, sourcegraph-class intelligence. Code is text-with-annotations. *[Phase A change 4 — optional hexagonal tagging is the only addition to this section]*

### Render pipeline

`traceability-render <module-id>` produces a human-scoped doc for one module: REQs, DESs, tests, code locations, with anchor links. Highlights orphan code and anaemic-context violations. Authoritative source remains markdown; rendered doc is derivative.

### Skills shipped (Assured bundle, in addition to Method 1's skills)

(Phase A changes 3, 5, 6 incorporated.)

- `commission-assured` — installs Assured (which includes Programme)
- `req-add`, `req-link` — convenience helpers for ID minting
- `code-annotate <artifact-id>` — auto-generates boilerplate annotations *[Phase A change 3; Line 2 §3 Claim 5]*
- `module-bound-check` — runs decomposition validators
- `kb-codeindex` — parses annotations, emits code-index
- `change-impact-annotate <artifact-id>` — guides team through change-impact declaration *[Phase A change 5]*
- `traceability-render <module-id>` — module-scoped human doc with link rendering
- `traceability-export <format>` — standard-specific formats: `do-178c-rtm`, `iec-62304-matrix`, `iso-26262-asil-matrix`, `fda-dhf-structure` *[Phase A change 6; Line 1 §3 Claim 6]*

### Commissioning guidance (new subsection)

(Phase A change 8.)

The `commission-assured` skill includes explicit guidance:

- **Defer decomposition.** Default `P1.SP1.M1.*`; refactor when operational pain motivates. *[Line 2 §3 Claim 7]*
- **Granularity per module.** Commissioning asks regulatory context; proposes granularity. Object-code is opt-in only.
- **Change-impact gating.** Enabled by default for IEC 62304 / FDA / ISO 26262 ASIL C/D.
- **Module-size heuristic.** "Scope of responsibility for one team (or part of one team)."
- **Programme / sub-program scale.** Coherent product / capability area / domain language; refactor over time.

## 5. Decomposition primitive — final call

(From `research/sdlc-bundles/synthesis/overall-scope-update.md` Section 5; library/decomposition-ddd-bazel.md preserves the citations.)

**DDD bounded contexts as primary decomposition unit, layered with Bazel's visibility-rule discipline, with optional hexagonal architecture for within-module structure.**

DDD is the highest-translatability pattern for a markdown-driven, filesystem-first framework: pure code organisation and team discipline, no build system or runtime required. Bazel's visibility-rule philosophy complements DDD without requiring Bazel itself. Hexagonal is opt-in for teams wanting further within-module isolation.

**Failure mode designed against**: anaemic contexts — business logic scattered across modules rather than concentrated in domain models. Detected by code-to-module assignment validators and orphan-code detectors; surfaced via `traceability-render`.

**Commissioning guidance emphasises deferral**: decomposition reflects current business domains, not speculative future architecture.

**Patterns explicitly NOT borrowed**: Erlang/OTP (needs runtime isolation), MBSE/SysML (needs specialised tools), ARINC 653 (needs RTOS), AUTOSAR (needs code generators), Bazel's full BUILD-graph (overkill).

## 6. Traceability rigour calibration — final call

(From overall-scope-update.md Section 6; library/regulatory-traceability-baseline.md preserves the citations.)

The Assured bundle natively supports the strictest standard's baseline (ISO 26262: bidirectional traceability at requirement-level granularity, with change-impact assessment, across all ASIL levels). This is simultaneously compatible with DO-178C, IEC 62304, IEC 61508, FDA 21 CFR Part 820.

**Native support**: bidirectional link integrity, granularity declaration, change-impact annotation, static build-time link validation, annotation-driven code linking, structured review records.

**Extension-point support**: standard-specific export templates (`traceability-export <format>`), change-impact gating (configurable), decomposition refactoring without ID loss, hexagonal architecture tagging.

**Explicit non-promises**: automatic change-impact detection (no AST analysis), traceability-link meaningfulness (humans verify; framework checks syntax), real-time database queries, identity-verification / approval-chain workflows at the schema level.

**Out of scope**: certification itself. The Assured bundle produces substrate that helps reach assurance; certification requires an accredited authority.

## 7. sphinx-needs adoption boundary

(From overall-scope-update.md Section 7; library/sphinx-needs-adoption-boundary.md preserves the citations.)

**Patterns we adopt**: declarative need-type schema, static build-time link validation, file-based Git-native storage, annotation-driven code parsing, modular per-module rendering.

**Patterns we explicitly do not adopt**: flat identifier scheme (REQ_001) — replaced by namespaced `P1.SP2.M3.REQ-007`; no built-in decomposition primitive (sphinx-needs hits scaling limits at 500+ pages) — Method 2 adds the registry; dynamic computed-field functions (steep learning curve, debugging-hostile); multiprocessing for 25K+-page projects (build times >30 min, memory >32 GB) — Method 2 targets ~5K specs per module; rich-text field support (round-trip lossy); Sphinx-based HTML rendering (markdown indices + optional `traceability-export` instead).

**Honest reading**: Method 2 ≈ "sphinx-needs + namespaced IDs + explicit decomposition primitive + agent-orchestrated retrieval/synthesis". The agent-orchestrated layer is our differentiator.

## 8. What is explicitly out of scope

(METHODS.md Section 6 with Phase A change 9 incorporated.)

- AST-level code intelligence
- IDE integration
- Full ALM database
- **Industry certification itself** — substrate, not certification *[Phase A change 9 strengthens this]*
- **Bidirectional ReqIF sync** — one-way export only (plugin-shaped) *[Phase A change 9 adds this]*
- Migration of pre-Assured projects' un-annotated code
- Decomposition suggestion
- Automatic semantic change-impact detection
- Distributed multi-team ID coordination
